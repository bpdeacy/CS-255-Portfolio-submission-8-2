# CS-255-Portfolio-submission-8-2

This project involved designing a system for DriverPass, a company that helps students prepare for driving tests. The system was built to support online training, practice exams, and scheduling for driving lessons, while also handling multiple user roles like customers, staff, and administrators.

One thing I did well was turning the client’s needs into clear requirements and system designs. The business requirements document defines what the system needs to do, and the system design document shows how it works through diagrams and technical details. Together, they provide a clear view of both the system’s purpose and structure.

If I could revise one part of my work, I would improve the system description in the presentation by more clearly outlining what the system does and what each user can do before going into details.

I interpreted user needs by working through the interview transcript and identifying what each user needed from the system. Those needs were then translated into requirements and design elements. This is important because the system has to solve real problems for the users, not just function correctly.

My approach to software design is to break the system into smaller parts, define requirements first, and then model workflows and structure using diagrams. Going forward, I would continue using these techniques while improving how I communicate designs to nontechnical audiences.
